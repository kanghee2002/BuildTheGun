# CommitGuide — AI 에이전트용

이 문서는 **사용자가 커밋 메시지 초안·작업 요약을 요청할 때** 에이전트가 따를 규칙이다.

## 할 일

1. **변경분을 직접 확인한다.**  
   저장소에서 `git diff` 또는 `git diff --staged`(이미 스테이징된 경우)를 실행해, 사용자가 말한 범위(또는 전체 작업 트리)의 실제 diff를 본다. 추측하지 않는다.

2. **한 커밋에 해당하는 작업 단위를 식별한다.**  
   diff만 보고 “이번에 묶을 수 있는 하나의 논리적 변경”이 무엇인지 정한다. 목적이 다른 변경(예: 문서 + 버그수정)이 섞여 있으면 **커밋을 나누는 것이 맞다**고 짚어 주고, 가능하면 메시지도 각각 제안한다.

3. **사용자에게 먼저 작업 내역을 요약해 제시한다.**  
   무엇이 바뀌었는지(파일 나열이 아니라 의미 중심) 짧게 정리한 뒤, 그 요약과 일치하는 커밋 제목·본문을 제안한다.

## 출력 형식

- **제목:** 영어, 한 줄. `타입: 주제`  
  - 콜론 뒤 공백 한 칸. 짧고 한 가지 주제만 담는다.  
  - **`타입:` 뒤 주제(영어)는 5단어 이내를 권장한다.** 타입 접두어는 단어 수에 넣지 않는다. 더 길어지면 압축하거나, 세부는 본문 첫 불릿으로 옮긴다.  
  - **주제의 첫 글자는 대문자**로 시작한다(타입 키워드 `feat` 등은 소문자 관례를 유지한다).  
  - **제목 끝에 마침표(`.`)를 붙이지 않는다.**  
  - **명령형(imperative)** 으로 쓴다. **과거형·완료형 표현은 쓰지 않는다.** (예: `Add`, `Fix`, `Document` — `Added`, `Fixed`, `Documented` 지양.)

- **본문:** 한글. 각 줄은 **`- `(하이픈 + 공백)** 으로 시작하는 불릿.  
  - **4~5줄**을 목표로 한다(내용이 매우 단순하면 그 이하도 가능하나, 통상 4~5줄).  
  - **각 불릿 줄은 50자 이내로 쓴다.** 공백 포함. 앞의 `- ` 두 문자는 길이에 넣지 않고 **그 뒤 본문만** 센다. 넘치면 불릿을 나누거나 표현을 압축한다.  
  - 경로 나열만 하지 말고, **변경 내용·의도·영향**이 읽히게 쓴다.  
  - **문장을 `~한다`로 끝내지 않는다.** 각 불릿은 **체언·명사구·짧은 구**로 마무리해 **마지막이 단어(또는 짧은 명사절)로 끝나게** 쓴다(예: `… 문서 추가`, `… 반영`, `… 정리` — `… 추가한다`, `… 반영한다` 지양).  
  - **가독성을 우선**하고, **핵심만 남긴다.** 부연·중복·장황한 수식은 줄이거나 누락해 **요약 한 줄**로 압축한다.  
  - **기본은 한글**이되, **특정 단어는 영어**로 쓴다. 코드·문서·도메인에서 통용되는 **고유명사·기술 용어·타입명·저장소 표기와 동일한 용어**는 한글 직역 대신 **프로젝트와 같은 영어 단어**를 쓴다. (예: `Pipeline` 사용 — `배선` 사용하지 않음)

- **최종 제안 형태(복사용):** 커밋 제목·본문을 **표, 2열, “제목 / 본문” 필드식 표** 등으로 나누지 않는다. **아래 예시 절과 같은 줄글**로만 제시한다: 첫 줄에 제목 전체, 다음 줄부터 본문 불릿만 연속으로 적는다(Git 관례대로 제목과 본문 사이에 **빈 줄 한 줄**을 둔다).  
  - 마크다운 **코드 펜스 블록**으로 감싸 “한 번에 복사하기 좋은” 덩어리로 주는 것은 허용한다. 내용은 여전히 줄글 구조(예시와 동형)여야 한다.

## 타입 선택

| 타입 | 사용 시점 |
|------|-----------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `build` | 빌드 설정, 패키지 설치/삭제 |
| `chore` | 위 분류에 안 맞는 잡무 |
| `ci` | CI 설정 |
| `docs` | 문서만 |
| `style` | 포맷/스타일(동작 동일) |
| `refactor` | 동작 유지 구조 변경 |
| `test` | 테스트 코드 |
| `perf` | 성능 |

애매하면 가장 가까운 타입을 고르고, 필요하면 한 줄로 왜 그 타입인지 사용자에게 말한다.

## 예시 (이 형태를 맞출 것)

최종 복사용 제안도 **이와 동일한 줄글 레이아웃**(제목 → 빈 줄 → 본문 불릿)을 쓴다. **실제 제안 시에는 위 본문 규칙(특히 불릿당 50자 이내)** 을 지킨다.

```
docs: Design combat system architecture
- 전투 설계 문서 추가(개요·총기·효과·Pipeline)
- 총알/총기/유물 데이터 스키마 초안 추가
- 총알 합체 규칙, OnKill 트리거·“연발” 용어 정리
- GDD 빌드 분류·효과 지침(문턱 보상) 반영
- TechDesign.md 제거, 전투 문서 구조로 재정리
```

## 추가 지침

- 사용자가 특정 파일/커밋 범위만 지정했다면 그 범위의 diff에 맞춘다.
- 제목 주제가 5단어를 넘기거나 여전히 애매하면, 본문 첫 불릿에 맥락을 더한다.
- `git add -p`로 나누는 제안은 **여러 목적의 변경이 한 번에 섞여 있을 때**만 짧게 언급한다.
